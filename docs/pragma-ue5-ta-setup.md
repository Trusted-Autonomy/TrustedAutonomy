# TA Setup: Pragma + UE5 Project with BMAD Architecture Docs

How to initialize a Trusted Autonomy-governed workflow for a Pragma Engine backend project
(Kotlin) connected to a Unreal Engine 5 game, using Perforce for VCS and BMAD for
architecture documentation maintenance.

---

## Prerequisites

- [TA installed](https://github.com/Trusted-Autonomy/TrustedAutonomy/releases/latest) (macOS, Linux, or Windows)
- `ANTHROPIC_API_KEY` set in your environment
- `p4` CLI installed and on PATH (for Perforce projects)
- Python 3 available (for the vcs-perforce plugin)
- Pragma project cloned locally
- UE5 game project synced locally

---

## Part 1: Init the Pragma Project (Kotlin + GitHub)

> **Note**: `ta init --template pragma` is coming in v0.15.30.3. Until then, scaffold manually:

In your Pragma project root:

```bash
mkdir -p .ta

cat > .ta/workflow.toml << 'EOF'
[project]
name = "your-game-pragma"
language = "kotlin"

[submit]
adapter = "git"

[verify]
commands = ["./gradlew ktlintCheck test"]

# Block draft build if ktlint fails
[[validate]]
stage = "pre_draft_build"
commands = ["./gradlew ktlintCheck"]
on_failure = "block"

# Warn (don't block) if tests fail at apply time
[[validate]]
stage = "pre_draft_apply"
commands = ["./gradlew test"]
on_failure = "warn"

# Inject the game project's architecture doc as read-only context for every agent
[context]
external_docs = [
  "../your-game-ue5/ARCHITECTURE.md",
]
EOF
```

Add TA runtime state to your `.gitignore`:

```
.ta/staging/
.ta/goals/
.ta/drafts/
.mcp.json
```

Run `ta doctor` to verify configuration.

## Part 2: Set Up Perforce VCS (UE5 Game Project)

### Install the vcs-perforce plugin

**macOS / Linux:**
```bash
cp "$(ta --data-dir)/plugins/vcs/vcs-perforce" ~/.config/ta/plugins/vcs/vcs-perforce
chmod +x ~/.config/ta/plugins/vcs/vcs-perforce
cp "$(ta --data-dir)/plugins/vcs/vcs-perforce.toml" ~/.config/ta/plugins/vcs/vcs-perforce.toml
```

**Windows (PowerShell):**
```powershell
$pluginDir = "$env:APPDATA\ta\plugins\vcs"
New-Item -ItemType Directory -Force -Path $pluginDir | Out-Null
Copy-Item "C:\Program Files\TrustedAutonomy\plugins\vcs\vcs-perforce" $pluginDir
Copy-Item "C:\Program Files\TrustedAutonomy\plugins\vcs\vcs-perforce.toml" $pluginDir
```

### Configure your UE5 game project

```bash
mkdir -p .ta   # in your UE5 project root
```

```toml
# .ta/workflow.toml (UE5 game project)
[project]
name = "your-game-ue5"
language = "cpp"

[submit]
adapter = "perforce"
workspace = "alice-ue5-workspace"     # your P4CLIENT value
protected_paths = [
  "//depot/main/Source/...",
  "//depot/main/Config/...",
  "//depot/main/Content/...",
]

[verify]
commands = []   # UE5 build verification — add your build script if available

# Inject Pragma architecture doc as context for every agent
[context]
external_docs = [
  "../pragma-project/ARCHITECTURE.md",
]
```

Set Perforce environment variables (add to shell profile or CI secrets):

```bash
export P4PORT=ssl:perforce.yourstudio.com:1666
export P4USER=your_p4_username
export P4CLIENT=alice-ue5-workspace
```

Add to your `.p4ignore`:

```
.ta/
.mcp.json
ONBOARDING.md
```

Verify plugin is detected:

```bash
ta doctor
```

## Part 3: BMAD Architecture Discovery (Pragma Project)

Run the BMAD architect persona as a TA goal. TA stages the run, the agent works normally,
and you get a reviewable draft before anything is written.

```bash
# From your Pragma project root
ta run "As a software architect using BMAD methodology: analyze this Pragma Engine deployment and UE5 SDK integration. Produce a complete ARCHITECTURE.md covering: (1) active Pragma services and their configs, (2) UE5 Pragma SDK integration points — player auth, matchmaking, commerce, social, (3) Kotlin plugin extension points, (4) data flows between UE5 client and Pragma services, (5) known tech debt and open integration gaps. Read all Pragma config files, Kotlin plugin sources, and any UE5 SDK usage documentation. Produce docs/ARCHITECTURE.md in this project." --agent bmad
```

Review and apply:

```bash
ta draft view latest     # read the proposed ARCHITECTURE.md
ta draft apply latest --git-commit
```

## Part 4: Reconcile with Existing Partial Architecture Doc

After the BMAD run produces `docs/ARCHITECTURE.md`, reconcile it with your existing partial
doc (e.g. in the game project at `Docs/Architecture.md`).

```bash
# Chain a follow-up goal onto the same draft thread
ta run "Compare docs/ARCHITECTURE.md (just generated by BMAD) with ../your-game-ue5/Docs/Architecture.md (the existing partial architecture doc). Identify: (1) items in the existing doc that are missing from the new doc, (2) discrepancies where versions or names differ between the two, (3) items the new doc covers that the existing one does not. Produce a reconciled docs/ARCHITECTURE.md that incorporates correct information from both sources. Where there is genuine conflict, keep both versions and flag with a comment: <!-- CONFLICT: describe the discrepancy -->. Do not discard the existing doc's details — only override where the BMAD analysis confirmed them to be wrong. Also produce a ARCHITECTURE-DIFF.md summary listing every change made and why." --follow-up
```

Review the reconciliation diff:

```bash
ta draft view latest     # shows both ARCHITECTURE.md and ARCHITECTURE-DIFF.md
ta draft apply latest --git-commit
```

## Part 5: Cross-Project Document Linking

Both projects reference each other's architecture docs via `[context] external_docs` in their
`workflow.toml` (configured in Parts 1 and 2 above).

When `ta run` launches an agent in either project, the other project's `ARCHITECTURE.md` is
injected as read-only background context. The agent sees both docs without any manual steps.

For a shared canonical source of truth, you can keep a single `ARCHITECTURE.md` in a shared
location (Perforce depot path or shared git submodule) and point both projects at it:

```toml
# Both .ta/workflow.toml files
[context]
external_docs = [
  "//depot/shared/docs/ARCHITECTURE.md",   # Perforce depot path
  # or:
  "../shared-docs/ARCHITECTURE.md",        # relative path to shared repo
]
```

## Part 6: Automated Doc Maintenance at Milestones

No new `ta` commands required — use TA's existing workflow engine.

### Option A: pre_draft_apply validation (runs before every apply)

Add to both projects' `.ta/workflow.toml`:

```toml
[[validate]]
stage = "pre_draft_apply"
commands = [
  "ta run 'Review ARCHITECTURE.md against the code changes in the current draft. If any section is now stale, update it. Focus on: changed service configs, new integration points, removed features.' --agent bmad --headless --no-draft"
]
on_failure = "warn"
```

This runs the BMAD doc-update agent before every `ta draft apply`. If docs are current, the
agent makes no changes. If they're stale, it updates them as part of the same apply.

### Option B: milestone-complete workflow (run after each phase PR merges)

Create `.ta/workflows/milestone-complete.toml`:

```toml
name = "milestone-complete"
description = "Post-phase architecture doc maintenance"

[[steps]]
id = "update_pragma_arch"
type = "run_goal"
goal = "Review docs/ARCHITECTURE.md against the code merged in the last phase. Update any sections that are now stale — new services, changed integration points, removed features, version changes. Commit the update with message 'docs: update ARCHITECTURE.md for completed phase'."
agent = "bmad"

[[steps]]
id = "sync_cross_project"
type = "run_goal"
goal = "Read ../your-game-ue5/Docs/Architecture.md and docs/ARCHITECTURE.md. Identify any cross-project discrepancies introduced by the last phase — new Pragma services not reflected in the game doc, or game integration changes not reflected in the Pragma doc. Produce or update ARCHITECTURE-SYNC.md listing the discrepancies and recommended updates."
agent = "claude-code"
depends_on = ["update_pragma_arch"]
```

After each phase PR merges:

```bash
ta workflow run milestone-complete
```

### Option C: Release-triggered doc update (wired into release pipeline)

If you use TA's governed workflow or release pipeline, add a doc-update step to
`.ta/workflows/governed-goal.toml` (or your custom release workflow):

```toml
[[steps]]
id = "doc_maintenance"
type = "run_goal"
goal = "After this phase's changes are applied: check if ARCHITECTURE.md reflects the new code. Update stale sections. No changes needed if docs are current."
agent = "bmad"
depends_on = ["apply_draft"]
```

This triggers automatically at every governed workflow run without any new commands.

## Quick Reference

```bash
# Check TA is configured correctly
ta doctor

# Run BMAD architecture discovery
ta run "Produce ARCHITECTURE.md for this project..." --agent bmad

# View and apply draft
ta draft view latest
ta draft apply latest --git-commit

# Reconcile with existing doc (chained follow-up)
ta run "Reconcile ARCHITECTURE.md with existing Docs/Architecture.md..." --follow-up

# Run post-milestone doc maintenance
ta workflow run milestone-complete

# Check current plan status
ta plan status
```
